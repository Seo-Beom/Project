## 2. 🔵 핵심 피드백 & 조언

### 2-1. 설계 레벨을 명확히 정하라
- 설계를 **어느 레벨까지** 할 것인지 정하는 게 중요
- **최소한 컨트롤러 + 서비스 목록** 수준까지는 설계 필요
- 기능을 뽑을 때 **실제 구현 가능한 레벨의 기능들인지** 확인할 것
- 기능 단위로 컨트롤러를 묶고, 각 컨트롤러에 속한 서비스 목록을 정리할 것
- 예: 통합지도 관리 컨트롤러, 관리자 기능 컨트롤러, 일반 사용자 서비스 컨트롤러 등

### 2-2. 필수 설계 산출물 3가지
멘토가 강조한 **반드시 만들어야 할 설계 문서**:

| 산출물 | 설명 |
|--------|------|
| **① 배치 설계서** | 공공 API를 주기적으로 호출하는 스케줄러 설계 (배치 주기: 1시간/반나절/하루 등 결정 필요) |
| **② 화면 설계서(화면 정의서)** | 각 화면의 구성요소와 동작 정의 |
| **③ 서비스(기능) 설계서** | 컨트롤러/서비스 기능 목록 정리 |

> **설계 순서**: DB 설계(ERD) → 화면 → 배치 → 기능 서비스 목록 순으로 진행  
> "DB 설계를 **최우선**으로" — 테이블 설계가 먼저 되어야 의사소통 미스 방지

#### ⚡ 산출물 우선순위 조정 (시간 부족 시)
- 화면 설계서는 **기획서(PPT/피그마)로 이미 충분히 커버 가능** → 생략 가능
- **기능 설계서 + 배치 설계서** 정도만 해도 괜찮음
- 배치 설계서도 별도 문서 대신 **목록에 배치 주기/시간 등을 함께 기재**하면 생략 가능
- 핵심: **목록(기능/배치/화면)은 무조건 있어야 한다** — 중복 개발 방지 필수

### 2-3. 공공 API 데이터 관리 범위 결정
- 축제 데이터의 **모수(범위)**를 명확히 정할 것:
  - **Case A**: 공공 API 결과만 사용
  - **Case B**: 공공 API + 관리자 수기 입력 데이터 포함
- 팀의 결정: 공공 API 데이터와 수기 등록 데이터를 **분리 관리** → 멘토 동의 ("오히려 더 명확하다")
- 관리자 축제 등록 시 **지도 좌표(위도/경도) 입력 필드** 추가 필요

### 2-4. API 데이터 갱신 시 우선순위 규칙
> 만약 공공 API 데이터도 수정 가능하게 한다면:
- API 원본 → 관리자 수정 → API 재호출 시 **어느 것이 우선?**
- 관리자 수정본 우선 vs API 결과 자동 덮어쓰기 → **룰 정의 필요**
- 현재 팀 방향: API 데이터는 수정 불가, 수기 등록만 별도 관리 → 이 문제 해소됨

### 2-5. WBS를 설계 기준으로 세분화
- 기획/설계 완료 후 WBS에 **기능 목록, 배치 목록, 화면 목록별 세부 일정** 반영
- 뭉뚱그린 일정 ❌ → **세세하게 나눌수록** 지연 감지가 빠름
- 예: "지도 API 연동" 하나로 잡지 말고, 세부 단위로 쪼갤 것

### 2-6. 외부 API는 최대한 빨리 테스트하라
> ⚠️ **가장 중요한 조언 중 하나**
- 지도 API(카카오/네이버), 공공 데이터 API는 **핵심 개발 기간 초반에 반드시 먼저 테스트**
- 외부 API는 우리가 컨트롤할 수 없으므로, 문제 발견 시 빠르게 대안 전환 필요
- 나머지 기능은 노력으로 커버 가능하지만, 외부 API는 불가

### 2-7. 차별화 전략 확인
- 이미 유사한 공공 축제 사이트가 존재 (멘토가 직접 찾아서 슬랙에 링크 공유)
- **"우리만의 차별점"**을 명확히 할 것
- 팀 답변: **커뮤니티 기능** (댓글, 후기, 블로그형 게시판) → 멘토 동의

### 2-8. 테스트 관련 (참고용)

| 개념 | 설명 |
|------|------|
| **단위 테스트** | 개별 기능/화면을 하나씩 테스트 (예: 조회 기능만 테스트) |
| **통합 테스트** | 처음부터 끝까지 시나리오 기반 테스트 (로그인 → 조회 → 이미지 확인 등 연동 포함) |
| **테스트 시나리오/계획서** | 실무에서는 산출물로 작성하지만, 이번 프로젝트에서는 시간 부족으로 **생략 가능** |

> 단위 테스트 도구: JUnit, JBuilder 등 활용 가능

### 2-9. 기능 목록 도출 팁
- **실제로 개발한다고 생각**하고 기능을 뽑을 것
- **실제 구현 가능한 레벨**인지 확인
- **최대한 구체적**이면 구체적일수록 좋음
- 중간중간 완성된 것을 멘토에게 공유 → 계속 피드백 받을 것
- 처음부터 올바른 방향은 어렵다 → **고쳐 나가는 게 맞음**

## 3. 🟡 코드 리뷰 & 경험 공유 권장

- **주 1회** 코드 리뷰/개발 리뷰 권장
- 각자 사용한 API, 경험, 시행착오를 **능동적으로 공유**
- 1명의 경험 → 4~5명이 공유하면 **3~4배의 학습 효과**
- 진척률뿐 아니라 "이런 시도를 했는데 안 돼서 이렇게 바꿨다" 같은 경험 공유 중요

## 6. 💡 멘토 추가 제안

### 산출물 & 프로세스
- 실제 필드에서 사용하는 **산출물 자료** 당일 중 공유 예정 (요구사항 명세서, WBS, 테스트 시나리오 등)
- 실무 프로세스: RFP(제안요청서) → Proposal(제안서) → 요구사항 명세서 → 요구사항 추적표 → 설계 → 개발 → 테스트
- 1차 프로젝트에서 경험 쌓고, **2차 프로젝트에서 멘토가 공유한 양식/프로세스 적용** 권장

### AI 도구 활용 권장
- 코파일럿 등 **AI 개발 지원 도구** 사용 경험도 취업 시 큰 메리트
- AI를 만드는 것도 중요하지만, **AI를 활용한 개발 경험**도 어필 포인트
- 실무에서는 공개 AI(ChatGPT, Gemini 등)는 보안 문제로 사용 제한 → 회사별 자체 AI 솔루션 사용 (IBM: Watson 계열, 삼성: 가우스)

### WBS 발표 팁
- 발표 시 PPT 형태의 간략한 WBS보다 **엑셀 형태의 상세 WBS**를 보여주면 실무 경험자들에게 "제대로 했구나" 인상
- 목록별 담당자, 세부 일정이 포함된 WBS가 **진짜 WBS의 표본**

### 커뮤니케이션
- 슬랙으로 수시 질문 가능 (평일 퇴근 후 답변, 주말 제외)
- GitHub 초대 완료 → 코드/문서 리뷰 가능
- 취업/이력서/면접 관련 멘토링도 2차 프로젝트 이후 제공 가능 (개인 DM 가능)
- 다음 멘토링: 피드백 + 이후 각자 켜놓고 작업하며 질문하는 방식도 가능

### 발표 관련
- 각자 파트를 나눠서 발표하는 게 좋은 경험 (모두에게 어필 기회)
- 다만 발표 선호도는 팀원들과 상의하여 결정

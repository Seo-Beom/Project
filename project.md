## 1️⃣ 프로젝트 기획 및 설계

> 🎯 **목적**: 프로젝트의 방향을 정하고, 어떻게 만들지 밑그림을 그리는 단계

| # | 세부 과업 | 설명 |
|---|---------|------|
| 1-1 | 프로젝트 목표 및 요구사항 정의 | 무엇을 만들 것인가? 어떤 기능이 필요한가? 요구사항 정의서 작성 |
| 1-2 | WBS 수립 및 일정 계획 | 해야 할 일을 쪼개서 주차별/담당자별 일정을 세움 |
| 1-3 | UI/UX 화면 설계 및 프로토타입 개발 | 와이어프레임(화면 레이아웃)을 그리고, 간단한 프로토타입 제작 |
| 1-4 | 대규모 트래픽을 고려한 MSA 아키텍처 설계 | 서비스를 마이크로서비스로 분리하여 설계 (예: 인증 서비스, 세션 서비스, 투표 서비스) |
| 1-5 | 기술 스택 및 인프라 구성 정의 | 어떤 기술을 쓸지 결정 (IBM Cloud, AWS, OpenShift, Kubernetes 등) |

### 💡 교육 프로젝트 적용 포인트
- **1-1, 1-2, 1-3**은 반드시 해야 함 → 기획서, WBS, 와이어프레임은 필수 산출물
- **1-4** MSA는 전체 구현이 어려우므로 → 모놀리식으로 시작하되 "MSA로 분리할 수 있는 구조"로 설계
- **1-5** 기술 스택은 학원에서 배운 것 위주로 선정 (React, Spring Boot, MariaDB 등)

## 2️⃣ 데이터 관리 및 전처리

> 🎯 **목적**: 데이터를 어떻게 저장하고 관리할지 설계하는 단계

| # | 세부 과업 | 설명 |
|---|---------|------|
| 2-1 | 이벤트 및 참가자 데이터 모델링 | ERD 작성 — 어떤 테이블이 필요하고, 관계가 어떻게 되는지 설계 |
| 2-2 | 데이터 수집 및 전처리 파이프라인 구축 | 데이터가 입력되고 저장되는 흐름 설계 |
| 2-3 | RDBMS(PostgreSQL) 및 NoSQL(MongoDB, Redis) 설계 | 관계형 DB + 캐시/비정형 데이터 저장소 설계 |
| 2-4 | 트랜잭션 관리 및 데이터 암호화 적용 | 데이터 무결성 보장 + 비밀번호 등 민감 정보 암호화 |

### 💡 교육 프로젝트 적용 포인트
- **2-1**은 핵심 → ERD는 개발 전에 반드시 완성해야 함
- **2-3** PostgreSQL 대신 **MariaDB/MySQL**로 대체 가능 (학원 교육 기반)
- **2-3** MongoDB는 선택 사항 → Redis만 실시간 기능(투표, 캐싱)에 활용하면 충분
- **2-4** 비밀번호 암호화는 Spring Security의 BCryptPasswordEncoder로 간단히 적용

## 3️⃣ 백엔드 개발 (MSA 기반)

> 🎯 **목적**: 서버 측 비즈니스 로직과 API를 개발하는 단계

| # | 세부 과업 | 설명 |
|---|---------|------|
| 3-1 | Spring Boot 기반 RESTful API 개발 | 모든 기능의 백엔드 API를 REST 방식으로 개발 |
| 3-2 | GraphQL & REST API 설계 및 구현 | (선택) GraphQL은 복잡한 조회에 활용, 기본은 REST |
| 3-3 | 이벤트 등록 및 세션 관리 API 개발 | 핵심 비즈니스 로직 — 이벤트/세션 CRUD |
| 3-4 | WebSocket을 활용한 실시간 토론 및 투표 기능 구축 | 실시간 Q&A, 실시간 투표 기능 (STOMP 프로토콜) |
| 3-5 | Spring Security 기반 JWT 인증 및 OAuth2 적용 | 로그인/인증 시스템 (JWT 토큰 기반) |
| 3-6 | 서비스 간 통신 (gRPC, Kafka) 적용 | (선택) MSA 환경에서 서비스 간 메시지 통신 |

### 💡 교육 프로젝트 적용 포인트
- **3-1, 3-3, 3-5**는 필수 → REST API + CRUD + JWT 인증은 기본
- **3-4** WebSocket은 차별화 포인트 → HTTP 폴링으로 먼저 구현하고, 여유 시 WebSocket 업그레이드
- **3-2** GraphQL은 제외 추천 (학습 비용 대비 효과 낮음)
- **3-6** gRPC, Kafka는 제외 추천 (MSA 수준의 분리가 어려운 교육 프로젝트에서는 불필요)

## 4️⃣ 프론트엔드 개발

> 🎯 **목적**: 사용자가 보고 사용하는 화면을 개발하는 단계

| # | 세부 과업 | 설명 |
|---|---------|------|
| 4-1 | React & TailwindCSS 기반 프론트엔드 대시보드 개발 | 메인 페이지, 목록, 상세 페이지 등 전체 UI 구현 |
| 4-2 | 이벤트 등록, 세션 참여, 실시간 토론 UI/UX 개발 | 핵심 기능의 사용자 인터페이스 개발 |
| 4-3 | 실시간 차트 및 인포그래픽 기능 개발 | Chart.js/Recharts로 참여 현황, 투표 결과 등 시각화 |
| 4-4 | 다국어 지원 기능 개발 | (선택) 한국어/영어 언어 전환 |
| 4-5 | SNS 연동 기능 구현 (SNS 알림 및 공유) | (선택) 카카오/네이버 공유, 푸시 알림 |

### 💡 교육 프로젝트 적용 포인트
- **4-1, 4-2**는 필수 → React로 핵심 화면 구현
- **4-3** 차트는 1~2개만 핵심적으로 구현하면 충분 (Bar 차트 + Pie 차트 정도)
- **4-4** 다국어는 제외 추천 (시간 대비 효과 낮음)
- **4-5** SNS 연동은 제외 추천 (OAuth 설정 복잡)

## 5️⃣ CI/CD 및 배포

> 🎯 **목적**: 개발한 코드를 자동으로 빌드·배포하고, 운영 환경을 구축하는 단계

| # | 세부 과업 | 설명 |
|---|---------|------|
| 5-1 | CI/CD 파이프라인 구축 (Jenkins, ArgoCD 활용) | 코드 푸시 → 자동 빌드 → 자동 배포 |
| 5-2 | Docker 기반 컨테이너화 | 애플리케이션을 Docker 이미지로 패키징 |
| 5-3 | OpenShift를 활용한 컨테이너 오케스트레이션 | (선택) 컨테이너 관리 자동화 |
| 5-4 | API Gateway (Kong / Spring Cloud Gateway) 구성 | (선택) API 요청 라우팅, 인증, 모니터링 |
| 5-5 | 배포 후 모니터링 시스템 구축 (Prometheus & Grafana) | (선택) 서버 상태, 응답 시간, 에러율 시각화 |
| 5-6 | CDN 및 Cloud Storage 활용 성능 최적화 | (선택) 정적 파일 CDN 배포, S3 스토리지 활용 |

### 💡 교육 프로젝트 적용 포인트
- **5-1** Jenkins 대신 **GitHub Actions**로 간단히 구축 가능
- **5-2** Docker는 **Cloud Native 필수 요소** → 반드시 구현
- **5-3** OpenShift는 제외 추천 → Docker + AWS EC2 또는 Docker Compose로 대체
- **5-4** API Gateway는 제외 추천 → 단일 서버 구성이면 불필요
- **5-5** Prometheus/Grafana는 있으면 가산점이지만 필수는 아님
- **5-6** CDN은 제외 추천

## 6️⃣ 테스트 및 최적화

> 🎯 **목적**: 코드 품질을 확인하고, 성능을 개선하는 단계

| # | 세부 과업 | 설명 |
|---|---------|------|
| 6-1 | 대규모 트래픽 대응을 위한 부하 테스트 및 성능 테스트 진행 | JMeter 등으로 동시 접속 테스트 |
| 6-2 | 보안 점검 및 API Rate Limiting 적용 | SQL Injection, XSS 등 보안 취약점 점검 |
| 6-3 | 서버 로드 밸런싱 및 트래픽 분산 처리 | (선택) 로드 밸런서 구성 |
| 6-4 | 데이터베이스 및 API 성능 최적화 | 쿼리 최적화, 인덱싱, N+1 문제 해결 |

### 💡 교육 프로젝트 적용 포인트
- **6-1** 간단한 부하 테스트라도 해보면 발표 때 어필 포인트
- **6-2** 기본 보안(XSS 방지, CORS 설정)은 적용 추천
- **6-3** 로드 밸런싱은 제외 추천
- **6-4** 쿼리 최적화, 인덱싱 정도는 적용 추천

| # | 세부 과업 | 설명 |
|---|---------|------|

- **7-1, 7-3** README.md로 아키텍처 + 사용법을 합쳐서 작성하면 효율적

## 🎯 교육 프로젝트 기준: 반드시 해야 할 것 vs 선택 사항

### ✅ 반드시 해야 할 것 (Must)

| 단계 | 과업 |
|------|------|
| 기획 | 요구사항 정의서, WBS, 와이어프레임 |
| 설계 | ERD, API 설계서, 화면 설계서 |
| 백엔드 | Spring Boot REST API, JWT 인증, 핵심 CRUD |
| 프론트엔드 | React 기반 UI, 핵심 화면 구현 |
| 배포 | Docker 컨테이너화 |
| 문서 | Swagger API 문서, 발표 PPT |

### 🔶 하면 가산점 (Should)

| 단계 | 과업 |
|------|------|
| 백엔드 | WebSocket 실시간 기능 |
| 프론트엔드 | 차트/인포그래픽 (1~2개) |
| 배포 | CI/CD 파이프라인 (GitHub Actions) |
| 테스트 | 기본 보안 점검, 쿼리 최적화 |

### ❌ 제외 추천 (Won't)

| 과업 | 제외 사유 |
|------|----------|
| GraphQL | 학습 비용 대비 효과 낮음 |
| gRPC / Kafka | MSA 수준 분리가 어려움 |
| OpenShift / Kubernetes | 인프라 학습 비용 큼 |
| Zoom 연동 | IBM 강사도 제외 권고 |
| 다국어 지원 | 시간 대비 효과 낮음 |
| SNS OAuth 로그인 | 설정 복잡, 핵심 아님 |

## 📊 7단계 vs 4주 일정 매핑

```
                    1주차        2주차        3주차        4주차
                 ┌──────────┬──────────┬──────────┬──────────┐
 1단계 기획/설계  │████████  │          │          │          │
 2단계 데이터     │████████  │          │          │          │
 3단계 백엔드     │          │████████  │████████  │          │
 4단계 프론트     │          │████████  │████████  │          │
 5단계 CI/CD     │          │          │          │████████  │
 6단계 테스트     │          │          │          │████████  │
 7단계 문서/발표  │          │          │          │████████  │
                 └──────────┴──────────┴──────────┴──────────┘
```

| 주차 | IBM 7단계 매핑 | 핵심 산출물 |
|------|--------------|-----------|
| **1주차** | 1단계(기획) + 2단계(데이터) | 기획서, ERD, 와이어프레임, WBS, API 설계서 |
| **2주차** | 3단계(백엔드) + 4단계(프론트) 시작 | DB 구축, 핵심 API, 프론트엔드 레이아웃 |
| **3주차** | 3단계(백엔드) + 4단계(프론트) 완성 | API 연동, 실시간 기능, 관리자 페이지 |
| **4주차** | 5단계(배포) + 6단계(테스트) + 7단계(文书) | Docker 배포, 테스트, Swagger, 발표 PPT |

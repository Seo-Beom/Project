## 1. 개요

### 1-1. 목적
이 문서는 공공데이터포털(data.go.kr)의 **한국관광공사 국문 관광정보 서비스 API**를 활용하여 축제/행사 데이터를 주기적으로 수집·갱신하는 배치 프로세스의 설계를 정의합니다.

### 1-2. 용어 정의

| 용어 | 설명 |
|------|------|
| **배치(Batch)** | 사용자 요청 없이 시스템이 자동으로 주기적으로 실행하는 작업 |
| **동기화(Sync)** | 외부 API의 데이터를 우리 DB에 반영하는 작업 |
| **Upsert** | 데이터가 없으면 INSERT, 있으면 UPDATE하는 작업 |
| **contentId** | 한국관광공사 API에서 각 관광 콘텐츠를 식별하는 고유 ID |
| **contentTypeId** | 관광 타입 ID (축제/공연/행사 = `15`) |

### 1-3. 사용 API 정보

| 항목 | 내용 |
|------|------|
| **API명** | 한국관광공사_국문 관광정보 서비스_GW (KorService1) |
| **제공처** | 공공데이터포털 (data.go.kr) |
| **Base URL** | `https://apis.data.go.kr/B551011/KorService1` |
| **인증 방식** | 서비스 키 (serviceKey) — 쿼리 파라미터 방식 |
| **응답 형식** | JSON (`_type=json` 파라미터 추가) |
| **일일 트래픽 제한** | 일반 인증키 기준 1,000건/일 (활용 신청 시 확인 필요) |

#### Festival 테이블 매핑

#### status 계산 로직

```java
// Java 비즈니스 로직
public FestivalStatus calculateStatus(LocalDate startDate, LocalDate endDate) {
    LocalDate today = LocalDate.now();
    
    if (today.isBefore(startDate)) {
        return FestivalStatus.UPCOMING;    // 예정
    } else if (!today.isAfter(endDate)) {
        return FestivalStatus.ONGOING;     // 진행 중
    } else {
        return FestivalStatus.ENDED;       // 종료
    }
}
```

### 3-5. Upsert 전략

```
┌──────────────────────────────────────────────────────────────┐
│  Upsert 판별 로직                                             │
│                                                              │
│  source_id(= API contentId) 기준으로:                         │
│                                                              │
│  1. DB에 해당 source_id가 없으면 → INSERT (신규 축제)          │
│                                                              │
│  2. DB에 해당 source_id가 있으면:                              │
│     → API의 modifiedtime과 DB의 api_modified_at 비교          │
│     → 다르면 UPDATE (변경된 필드만 덮어쓰기)                    │
│     → 같으면 SKIP (갱신 불필요)                               │
│                                                              │
│  ⚠️ 주의: source='API'인 데이터만 Upsert 대상                 │
│           source='MANUAL'(수기 등록)은 배치에서 건드리지 않음    │
└──────────────────────────────────────────────────────────────┘
```

### 3-6. 에러 처리

| 에러 유형 | 상황 | 처리 방법 |
|----------|------|----------|
| **API 호출 실패** | HTTP 4xx/5xx, 타임아웃 | 3회 재시도 (간격 5분), 실패 시 로그 기록 후 종료 |
| **API 응답 파싱 실패** | resultCode ≠ "0000", JSON 파싱 에러 | 해당 페이지 스킵, 에러 로그 기록, 다음 페이지 계속 |
| **개별 데이터 오류** | 필수 필드(title, contentid) 누락 | 해당 item 스킵, 에러 로그에 contentId 기록 |
| **DB 저장 실패** | SQL 에러, 커넥션 풀 고갈 | 해당 건 롤백, 에러 로그 기록, 나머지 계속 처리 |
| **트래픽 제한 초과** | 일일 API 호출 제한 초과 | 즉시 중단, 남은 건수 로그 기록, 다음 스케줄에 재시도 |

### 3-7. 페이징 전략

```
전체 데이터 조회 시 페이징 처리:

1. 첫 페이지(pageNo=1) 호출 → response.body.totalCount 확인
2. totalPages = ceil(totalCount / numOfRows) 계산
3. pageNo=1 ~ totalPages까지 순차 호출
4. 각 페이지 호출 사이 100ms~500ms 딜레이 (API 서버 부담 경감)

예시: totalCount=350, numOfRows=100 → totalPages=4
  - Page 1: item[0]~item[99]
  - Page 2: item[100]~item[199]
  - Page 3: item[200]~item[299]
  - Page 4: item[300]~item[349]
```

| **담당 클래스** | `FestivalStatusScheduler.java` (global/scheduler/) |

### 4-3. 실행 SQL

```sql
-- 상태가 변경이 필요한 축제만 업데이트 (변경 없는 축제는 건드리지 않음)
UPDATE festival
SET 
    status = CASE
        WHEN CURRENT_DATE < start_date THEN 'UPCOMING'
        WHEN CURRENT_DATE >= start_date AND CURRENT_DATE <= end_date THEN 'ONGOING'
        WHEN CURRENT_DATE > end_date THEN 'ENDED'
    END,
    updated_at = NOW()
WHERE status != CASE
        WHEN CURRENT_DATE < start_date THEN 'UPCOMING'
        WHEN CURRENT_DATE >= start_date AND CURRENT_DATE <= end_date THEN 'ONGOING'
        WHEN CURRENT_DATE > end_date THEN 'ENDED'
    END;
```

### 5-4. 중복 실행 방지

```java
// AtomicBoolean 또는 Redis 플래그로 동시 실행 방지
private final AtomicBoolean isSyncing = new AtomicBoolean(false);

public SyncResult syncFestivals(SyncCommand command) {
    if (!isSyncing.compareAndSet(false, true)) {
        throw new BusinessException(ErrorCode.SYNC_ALREADY_IN_PROGRESS);
    }
    try {
        // 동기화 로직 실행
        return doSync(command);
    } finally {
        isSyncing.set(false);
    }
}
```

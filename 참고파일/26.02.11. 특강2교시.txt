============================================================
  IBM 박성식 특강 요약 (17:30 이후)
  - 프로젝트 수행 방법론 및 단계별 가이드
============================================================


■ 개요
──────────────────────────────────────
- 전통적인 워터폴(Waterfall) 방식 기반의 프로젝트 수행 방법 설명
- 참고: 요즘은 애자일(Agile) 등 다른 방법론도 사용되지만,
  전통적인 방법도 나쁘지 않음
- 프로젝트는 크게 5단계로 진행:
  ① 프로젝트 준비 → ② 요구사항 정의 → ③ 설계 → ④ 개발 → ⑤ 테스트


============================================================
  1단계: 프로젝트 준비
============================================================

■ 목적
- 프로젝트 시작 전 환경 준비, 팀 구성, 범위 설정

■ 주요 작업
- 프로젝트 범위 정의 (하이레벨 스코프 설정)
- 프로젝트 팀 구성 및 역할 분담
- 프로젝트 일정 수립
- WBS(Work Breakdown Structure) 작성
  · 해야 할 태스크를 세분화
  · 각 태스크별 일정, 담당자, 산출물 정의
  · 단계별 진행 관리

■ 핵심 조언
- 일정 관리가 매우 중요 (보통 프로젝트 기간은 넉넉하지 않음)
- 부족한 시간 내에 태스크를 적절하게 배분하고 리소스를 배치하는 것이 핵심
- 가능하면 상세하게 일정을 잡을 것을 추천


============================================================
  2단계: 요구사항 정의
============================================================

■ 목적
- 어떤 기능들을 만들 것인가를 명확하게 파악하는 단계
- 사용자의 요구사항을 받고 정리하는 단계

■ 주요 작업
- 이해관계자 식별 및 인터뷰 진행
- 현행(As-Is) 시스템 분석
  · 기존 시스템의 문제점 파악
  · 어떤 기능을 계승하고 어떤 기능을 버릴지 판단
- 초기 시스템 목표 설정 (To-Be 방향 정의)
- 요구사항 정의서 작성
  · 기능적 요구사항: 코딩으로 구현할 어플리케이션 기능들
  · 비기능적 요구사항: 성능, 보안, 안정성, 가용성, 확장성,
    인터페이스, 화면 요구사항 등
- 요구사항 우선순위 선정
  · Must (반드시 해야 하는 것)
  · Should (해야 하지만 우선순위 조정 가능)
  · Could (좋지만 필수는 아닌 것)
  · Won't (할 필요 없는 것)
- 요구사항 확정 (검토를 통한 최종 구현 범위 결정)

■ 핵심 조언
- 요구사항 정의는 프로젝트에서 가장 중요한 단계 중 하나
- 요구사항이 모호하거나 명확하지 않으면 프로젝트가 흔들림
- 가장 정확하고 상세하게 정리되어야 함
- 범위를 넘어서는 요구사항이 들어오면 일정에 맞추기 어려워짐
- 팀원들끼리 잘 회의해서 기능 범위를 명확하게 정할 것


============================================================
  3단계: 설계
============================================================

■ 목적
- 정의된 기능들을 어떻게 만들 것인가를 결정하는 단계
- 설계 명세서/설계서 작성

■ 주요 작업

[아키텍처 설계]
- 전체 시스템 구성도 (컴포넌트, 서비스 간 구성)
- 클라우드 vs 온프레미스 vs 하이브리드 결정
- 네트워크, 보안 구조 설계
- 이중화 구조 (HA), 로드밸런싱, 캐시 구성
- 보통 프로젝트팀에서 아키텍트가 담당

[DB 설계]
- RDB 기반 테이블 설계
- ERD(Entity Relationship Diagram) 작성
  · 테이블 간 관계도 그리기
- 테이블 정의서 작성
  · 컬럼명, 타입, 키 등 속성 정의
- DB 설계는 매우 중요 (대부분의 기능이 DB 기반으로 출발)

[화면(UI) 설계]
- 와이어프레임 작성 (전체 흐름 파악)
- 화면 설계서 작성
  · 화면 구성, 버튼 기능, 레이블 등 상세 정의
- 사용자가 유일하게 확인할 수 있는 것이 화면이므로 매우 중요

[기능 설계]
- 비즈니스 로직 설계
- 배치(Batch) 프로그램 설계
  · 화면 없이 일괄 처리되는 작업 (예: 일일 결산)
  · 스케줄러를 통한 자동 실행

[API 설계]
- API 명세서 작성
  · Input/Output 정의
  · 프로토콜, 인증 방법 정의
- API 개발팀이 작성, 사용처에서 참고하여 사용

[인터페이스 설계]
- 타 시스템과의 연계 방법, 데이터 교환 방식 정의

[개발환경 및 배포 구성]
- 개발/테스트/운영 시스템 환경 구성
- CI/CD 파이프라인 구성
- 테스트 계획 수립

■ 핵심 조언
- 설계가 잘 되어야 개발을 잘 할 수 있음
- 설계 불량 → 변경/추가 작업 급증 → 일정 지연 → 품질 저하
- 무작정 개발하지 말고 설계를 먼저 해볼 것을 강력 추천
- 설계를 잘하려면: ①분석을 잘하는 것 ②경험치가 중요


============================================================
  4단계: 개발
============================================================

■ 목적
- 설계를 기반으로 실제 소프트웨어를 코딩하는 단계
- 안정적인 코드 품질 확보

■ 주요 작업

[기능 개발]
- 프론트엔드 화면 개발 (와이어프레임/화면설계서 기반)
- 백엔드 API 개발 (비즈니스 로직 구현)
- DB 테이블 생성 (설계된 ERD 기반)

[개발 순서 (일반적)]
1. DB 테이블 생성 (가장 먼저)
2. API 개발 + 화면 개발 (병행 진행)
3. 화면에 API 연동 후 최종 검증

[인터페이스 연동]
- 타 시스템과의 인터페이스 개발 및 기본 연동 테스트

[코드 리뷰]
- 시니어 개발자/파트 리더에 의한 코드 품질 검토
- 코딩 표준 준수 여부 확인
- 현실적으로 시간 부족으로 어려울 수 있지만, 프로젝트에서 해볼 것을 추천

[단위 테스트]
- 개발된 개별 기능에 대한 테스트
- 테스트 케이스 작성 후 실행
- JUnit 등 테스트 도구 사용 추천

■ 핵심 조언
- 개발 단계가 프로젝트의 최종 결과물을 만드는 가장 중요한 단계
- 프로젝트의 품질은 개발 단계에서 결정됨
- 통합 테스트를 위한 준비도 개발 단계에서 함께 진행


============================================================
  5단계: 테스트
============================================================

■ 목적
- 전체 기능을 연계하여 최종 품질 확인
- 시스템 오픈 가능 여부 판단
- 사용자/고객의 신뢰 확보

■ 주요 작업

[기능 테스트 (통합 테스트)]
- 테스트 계획 작성
- 테스트 시나리오/케이스 작성
  · 테스트 데이터 및 예상 결과 포함
  · 전체 기능 커버리지 확보
- 테스트 실행 및 결과 기록
- 통과 기준 설정 (예: 100개 케이스 중 100% 통과)
- 오류 심각도 분류
  · 심각한 오류: 반드시 수정 후 오픈
  · 마이너 오류: 오픈 후 처리 가능 여부 의사결정
- 무작위 테스트(탐색적 테스트)도 병행

[비기능 테스트]
- 성능/부하 테스트
  · 응답 시간, TPS(초당 처리 건수) 등 목표치 설정
  · 로드 테스트 도구를 사용하여 부하 주입 후 확인
- 보안 테스트
  · 소스코드 보안 취약점 분석 도구 활용
  · 모의 해킹(Penetration Testing) 수행
- 가용성 테스트
  · 페일오버(Failover) 테스트
  · 액티브-스탠바이 환경에서 서비스 연속성 확인
- 백업/복구 테스트
  · 백업 및 복구 절차 정상 동작 확인

■ 핵심 조언
- 테스트 단계를 잘 거치지 않으면 버그가 있는 상태로 오픈될 수 있음
- 특히 금전 처리 등 중요 시스템에서는 심각한 문제 발생 가능
- 최종 품질을 확인하는 마지막 관문이므로 철저히 수행할 것
- 테스트 결과 산출물을 만들어 보는 것도 추천


============================================================
  전체 요약 및 핵심 메시지
============================================================

1. 프로젝트 5단계: 준비 → 정의 → 설계 → 개발 → 테스트
2. 모든 단계가 중요하며, 각 단계가 다음 단계의 품질에 직접 영향
3. 요구사항 정의가 모호하면 → 설계 흔들림 → 개발 변경 급증 → 일정 지연
4. 설계 없이 바로 개발하지 말 것 (설계 문서 작성 추천)
5. WBS를 통한 일정 관리가 매우 중요
6. 범위를 현실적으로 축소하여 설정할 것
7. 가능하면 다음 산출물들을 작성해볼 것을 추천:
   - WBS (일정 관리)
   - 요구사항 정의서
   - 아키텍처 정의서
   - ERD 및 테이블 정의서
   - 화면 설계서 (와이어프레임)
   - API 명세서
   - 테스트 케이스 및 결과서
